# Portable Export Excludes Atomic-Publish Scratch Files

## Status

Accepted.

## Index Metadata

- Decision ID: AOA-SM-D-0019
- Original date: 2026-07-19
- Owner surfaces: `scripts/aoa_session_memory.py`, `INSTALL.md`, `tests/test_session_memory.py`, `docs/decisions/`
- Surface classes: portable export, concurrent generation, public-safe copy boundary
- Projection layers: portable kernel, generated navigation maps
- Guard families: atomic publish, transient-file exclusion, concurrent export, leakage prevention
- Posture: accepted

## Context

The live source root can publish generated navigation files atomically while a
portable export is reading authored kernel directories. Atomic writers create
hidden scratch files and replace the stable target after the write succeeds.
A recursive copier may enumerate one of those scratch files just before the
writer replaces it, causing an otherwise read-only export to fail. Copying the
scratch file when it still exists would be worse: it would expose incomplete
generated state as portable source.

## Options Considered

- Acquire the runtime maintenance lease and stop all generated writers during
  every export. Rejected because portable source export must not depend on a
  live runtime coordinator and the lease does not own every authored writer.
- Retry any disappeared path or silently ignore arbitrary copy errors.
  Rejected because a missing authored source file must remain a hard failure.
- Exclude only recognized hidden atomic-publish scratch names while copying
  stable authored targets normally.

## Decision

Portable recursive copy excludes hidden names containing the atomic scratch
marker `.tmp`, including SQLite journal derivatives of a scratch database.
These paths are never portable source and are not counted as copied artifacts.
Stable authored files, including other hidden authored files, continue through
the ordinary copy route, and any failure to copy them remains visible.

## Rationale

The stable post-replace target is the only published state. Excluding its
private scratch predecessor makes concurrent export restartable without
turning the portable copier into a mutation coordinator or weakening errors
for real source loss. It also prevents partial generated content and
host-process naming from leaking into a public-safe bundle.

## Consequences

- Portable export can overlap atomic navigation generation without racing a
  disappearing scratch path.
- Incomplete or uncommitted generated bytes cannot enter the bundle through an
  atomic temporary filename.
- A legitimately authored hidden file remains included unless it uses the
  reserved `.tmp` scratch marker.
- Producers must keep atomic scratch files hidden and marked with `.tmp`; a
  different transient convention requires an explicit owner update and test.

## Boundaries

This decision does not make a mutable source tree a coherent multi-file
snapshot, admit runtime databases or diagnostics, hide arbitrary copy errors,
or replace source/portable parity and leakage audits. It governs only
ephemeral atomic-publish scratch paths inside otherwise portable directories.

## Follow-Up Route

Run a clean portable export and standalone validation while normal generated
map writers remain enabled. Reopen this boundary if a producer uses a
different scratch convention or if a multi-file projection requires a
generation-manifest snapshot rather than independent atomic targets.

## Verification

A focused regression distinguishes an authored hidden file from representative
JSON and SQLite atomic scratch names. The end-to-end portable installation test
copies the live source root while concurrent generated-map publication is
allowed and then runs standalone validation.
